# 秒杀高并发专题学习记录

秒杀专题用于学习电商系统最极端的热点流量场景。

这块不要和普通下单混在一起改。普通下单重点是业务状态正确、库存扣减正确、订单数据完整；秒杀重点是削峰、限流、热点库存原子扣减、异步落库和最终一致性。

## 学习目标

围绕秒杀活动 / 秒杀库存 / 秒杀请求入口，学清楚：

```text
普通下单为什么不能直接扛秒杀
秒杀活动和商品库存怎么建模
Redis Lua 为什么适合扣热点库存
Redis 库存和 MySQL 库存怎么最终一致
请求削峰、异步下单和防重复下单怎么设计
```

## 当前进度

### 下一步：秒杀建模和链路边界

这一小步先不急着写 Redis Lua，也不急着改订单主链路。

本轮要先梳理：

- 秒杀活动表需要哪些字段。
- 秒杀商品 / 秒杀库存和普通商品库存是什么关系。
- 秒杀请求入口为什么应该独立于普通下单入口。
- 哪些校验可以在网关 / Redis 前置完成，哪些必须落到 MySQL。
- Redis 预扣库存成功后，订单落库失败时怎么补偿。

核心结论先定为：

```text
普通下单链路以 MySQL 为核心保证正确性；
秒杀链路以 Redis 扛热点，用异步化保护 MySQL；
最终仍然要以 MySQL 订单和库存结果作为持久化事实。
```

## 面试回答模板

如果面试问：

```text
秒杀系统怎么防超卖？
```

可以这样答：

```text
普通下单可以用 MySQL 条件更新扣库存，比如 stock >= buy_count；
但秒杀是热点商品，所有请求打到同一行 MySQL 会造成锁竞争。

秒杀会先把活动库存预热到 Redis，用 Lua 脚本把判断库存、扣减库存、防重复参与放在一个原子操作里。
Redis 扣减成功只表示拿到了秒杀资格，后面通过 MQ 或本地消息表异步创建订单。
订单和库存最终仍然要落到 MySQL，并通过超时取消、补偿任务、库存校准兜底。
```
