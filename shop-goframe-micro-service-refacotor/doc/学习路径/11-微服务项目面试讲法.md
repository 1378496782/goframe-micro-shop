# 微服务电商项目面试讲法

这份文档用于准备简历里的项目一：

```text
基于 GoFrame + gRPC 的微服务电商系统实践
```

目标不是背八股，而是形成一套稳定、可展开、能落到代码和链路的讲法。

## 一、项目总口径

### 30 秒版本

这个项目是我在单体电商项目基础上做的一次微服务化和工程化实践。系统按用户、商品、订单、互动、搜索、H5 网关和管理后台进行拆分，服务之间通过 gRPC 调用，通过 etcd 做服务注册发现。项目重点实践了订单最终一致性、RabbitMQ 基础异步消息、Outbox 本地消息表、Redis 商品详情缓存、Elasticsearch 商品搜索、评论点赞幂等和 Docker Compose 多服务部署。

### 1 分钟版本

这个项目主要解决三个问题。

第一是服务拆分和调用链路。我把用户、商品、订单、互动、搜索等能力拆成独立服务，前台请求统一经过 H5 网关，再由网关调用后端 gRPC 服务。

第二是交易链路的一致性。比如支付成功后增加商品销量、取消订单后恢复库存，这些不是简单同步调用就能处理好的，所以我引入了 RabbitMQ 基础异步消息、消费幂等、Outbox 本地消息表和补偿任务，保证主状态先落库，副作用失败后可以重试补偿。

第三是读性能和工程部署。商品详情做了 Redis Cache Aside、空值缓存、TTL 随机化和互斥重建；商品搜索用 MySQL Binlog / CDC 同步到 Elasticsearch；最后用 Docker Compose 把 MySQL、Redis、RabbitMQ、etcd、ES、业务服务和管理后台一起编排起来，做了本地生产形态验证。

### 3 分钟版本

可以按这个顺序讲：

```text
背景 -> 架构 -> 核心链路 -> 一致性设计 -> 缓存/搜索/互动 -> 部署验证 -> 收获
```

推荐表达：

这个项目是一个 GoFrame 微服务电商系统，定位是把我之前做过的单体电商后端进一步拆成多服务协作形态。整体服务包括 H5 网关、用户服务、商品服务、订单服务、互动服务、搜索服务、轮播服务和管理后台。

请求入口主要是 H5 网关。比如商品列表、购物车、订单、评论、点赞、搜索这些前台接口都从 `gateway-h5` 进入，然后通过 gRPC 调用后端服务。服务注册发现用 etcd，避免网关写死后端实例地址。

我重点做的是订单链路。创建订单时，订单服务会调用商品服务获取购物车商品快照，扣库存后创建订单主表和订单明细。取消订单时，先校验订单归属和订单状态，然后把待支付订单改成已取消，再异步恢复库存。支付成功后，订单主状态先从待支付变成已支付，增加销量这种副作用通过 RabbitMQ 事件驱动执行。

为了处理异步链路的不确定性，我加了 `sales_status` 副状态和补偿任务。消费者处理支付成功事件时，不是直接增加销量，而是先用状态字段抢占，比如 `未同步 -> 同步中`，成功后改为已同步，失败后改为同步失败。后台补偿任务会扫描同步失败或者长时间卡在同步中的订单，再重新尝试增加销量。这样可以应对 MQ 消费失败、服务重启、重复消息等情况。

另外我还做了 Outbox 本地消息表，用于解决“订单本地事务成功，但 MQ 发布失败”的一致性缺口。核心思路是业务事务里先把待投递事件写到本地消息表，后续由中继任务投递 MQ，投递成功再更新消息状态。这样消息发布失败不会直接丢事件。

商品详情这块用了 Redis Cache Aside：先查缓存，未命中查 MySQL，再写缓存。为了防缓存穿透，不存在的商品也写短 TTL 空值；为了防雪崩，TTL 加随机抖动；为了防热点 key 击穿，单实例下用了本地互斥锁控制回源。

搜索链路是 MySQL 到 ES。商品数据仍以 MySQL 为准，search-service 通过 Binlog / CDC 监听商品表变化，同步到 Elasticsearch。搜索时用 multi-match 查询商品名、品牌、标签和详情，品牌和价格这类条件用 filter。

互动模块做了评论、回复、点赞和收藏。评论模型里用了 `parent_id / root_id / reply_user_id` 来区分直接回复对象、所属一级评论和被回复用户。点赞用 `user_id + type + object_id` 唯一索引防重复，重复点赞和重复取消都按幂等成功处理。评论点赞计数放在本地事务里和用户行为记录一起更新，避免记录和计数不一致。

最后我用 Docker Compose 编排了 MySQL、Redis、RabbitMQ、etcd、Elasticsearch、H5 网关、业务服务和管理后台，验证了管理后台、H5 网关、搜索服务、RabbitMQ 消费链路和用户注册发券链路。

## 三、架构类高频问题

### Q1：这个项目整体架构是什么？

回答：

整体是 HTTP 网关 + gRPC 微服务 + 基础中间件的结构。

前台请求先进入 `gateway-h5`，网关负责 HTTP 入参、登录态解析、统一响应和调用后端 gRPC 服务。后端服务按业务拆成用户、商品、订单、互动、搜索、轮播等服务。服务注册发现使用 etcd。数据层主要是 MySQL，Redis 做缓存和分布式锁辅助，RabbitMQ 做基础异步消息，Elasticsearch 做商品搜索读模型。

可以补一句：

我这个项目不是为了追求复杂拆分，而是围绕电商核心链路拆分：商品、订单、用户、互动这些边界比较清晰，搜索单独拆出来是因为它读 ES，和交易库的读写模型不同。

### Q2：为什么要用 H5 网关？

回答：

网关主要解决三个问题。

第一，统一对外暴露 HTTP 接口，前端不需要直接感知后端有多少个 gRPC 服务。

第二，做横切逻辑，比如登录态解析、参数转换、统一响应、错误处理。

第三，聚合后端服务能力。比如订单预览接口，网关只对前端暴露一个 HTTP 接口，内部由 order-service 编排 goods-service 获取购物车商品快照。

不要把网关说成复杂 API Gateway。当前项目里的 `gateway-h5` 更像业务 HTTP 网关。

### Q3：为什么用 gRPC？

回答：

服务拆分后，服务之间需要稳定的接口协议。gRPC 的好处是 IDL 明确、请求响应结构清晰、性能也比较适合服务间调用。对这个项目来说，gRPC 最大价值不是性能，而是让服务间接口从普通函数调用变成明确的 RPC 契约。

补充：

比如 H5 网关调用订单服务，不直接依赖订单服务内部实现，而是依赖 protobuf 定义的请求响应结构。

### Q4：etcd 在项目里干什么？

回答：

etcd 用于服务注册与发现。每个 gRPC 服务启动后把自己的服务名和地址注册到 etcd，调用方通过服务名发现可用实例。这样调用方不用写死某个 IP 和端口，后续多实例部署时也更容易扩展。

当前项目里主要用于本地多服务编排下的服务发现，没有深入做复杂的健康检查、权重、灰度等治理能力。

## 四、订单链路高频问题

### Q5：创建订单的完整链路是什么？

回答：

创建订单链路是：

```text
gateway-h5
  -> order-service CreateFromCart
  -> goods-service GetSelectedItems
  -> goods-service DeductStock
  -> order-service 创建订单主表
  -> order-service 创建订单商品快照
  -> goods-service DeleteSelectedItems
```

关键点：

1. 用户身份从 token / context 里取，不信前端传 `user_id`。
2. 价格不信前端传，订单服务从商品服务获取购物车商品快照。
3. 扣库存用条件更新，比如 `stock >= buy_count`。
4. 订单明细保存下单时的商品快照，避免后续商品改价影响历史订单。
5. 删除购物车是跨服务后置动作，不能放进 order-service 本地事务里。

### Q6：扣库存怎么防止超卖？

回答：

普通下单场景里，用 MySQL 条件更新控制：

```sql
UPDATE goods_info
SET stock = stock - ?
WHERE id = ? AND stock >= ?
```

只有 `RowsAffected = 1` 才说明扣库存成功。如果库存不足，更新影响行数为 0，订单创建失败。

多商品下单时，可以先按 `goods_id` 排序再扣库存，减少并发事务里不同顺序加锁导致的死锁风险。

### Q7：创建订单失败时怎么回滚库存？

回答：

如果扣库存成功，但订单后续创建失败，需要按订单商品明细或扣减结果恢复库存。这里要区分：

普通下单链路核心是 MySQL 条件更新和补偿。

如果扣库存和订单写入跨服务，无法用一个本地事务包住所有操作，所以要接受“局部失败后补偿”的设计。更完整的做法是把库存扣减、订单创建、购物车清理这些动作拆成明确状态，并通过补偿任务或事件机制兜底。

### Q8：取消订单为什么要异步恢复库存？

回答：

取消订单时，订单主状态是最重要的事实。用户取消成功后，订单应该先从待支付变为已取消。恢复库存属于取消后的副作用。

如果恢复库存同步调用失败，不应该把订单状态重新改回待支付，否则用户视角会混乱。更稳的是保留已取消状态，让恢复库存动作通过消息重试或补偿任务兜底。

### Q9：订单状态机怎么设计？

回答：

当前项目里先收敛了核心状态：

```text
待支付 -> 已支付
待支付 -> 已取消
```

更新状态时不直接 `UPDATE status = newStatus`，而是带上 `fromStatus -> toStatus` 条件。

比如：

```sql
UPDATE order_info
SET status = paid
WHERE id = ? AND status = pending
```

这样支付回调、用户取消、超时取消并发时，只有一个流程能抢到状态流转，其他流程看到 `RowsAffected = 0` 就知道订单已经被处理过。

### Q10：`sales_status` 是什么，为什么不直接放到订单 status 里？

回答：

`status` 是订单主状态，比如待支付、已支付、已取消。它描述交易事实。

`sales_status` 是支付成功后“销量同步”这个副作用的状态，比如未同步、同步中、已同步、同步失败。

两者不能混在一起。支付成功就是支付成功，不能因为增加销量失败就把订单状态回滚。增加销量失败应该由副状态记录，再由补偿任务继续处理。

## 五、RabbitMQ 与 Outbox 高频问题

### Q11：你项目里 RabbitMQ 用在哪里？

回答：

主要用在订单支付成功后的异步副作用，以及用户注册后发券这类事件驱动场景。

以支付成功加销量为例：

```text
order-service 支付成功
  -> 发布 OrderPaidEvent
  -> OrderPaidSalesConsumer 消费事件
  -> 调 goods-service 增加销量
  -> 更新 sales_status
```

我这里用的是 RabbitMQ 的基础能力：exchange、routing key、queue、consumer、ack，以及业务侧消费幂等。

### Q12：为什么支付成功后不直接同步增加销量？

回答：

支付成功是主链路事实，增加销量是支付成功后的副作用。

如果同步调用 goods-service 增加销量，goods-service 一旦超时或失败，支付回调链路会被拖慢，甚至影响支付平台回调返回。但实际上订单已经支付成功，不应该因为销量同步失败影响支付状态。

所以我把增加销量改成异步事件：订单状态先落库，支付回调尽快返回；销量同步失败通过 `sales_status` 和补偿任务兜底。

### Q13：RabbitMQ 重复投递怎么办？

回答：

我按“消息可能至少一次投递”来设计消费者，所以消费者必须幂等。

在支付成功加销量场景里，消费者不是直接增加销量，而是先抢占 `sales_status`：

```text
0 未同步 -> 3 同步中
```

只有抢占成功的消费者才继续调用 goods-service 增加销量。如果消息重复投递，后面的消费者发现状态已经是同步中或已同步，就不会重复增加销量。

### Q14：消费失败怎么办？

回答：

分两层处理。

第一层是消费者处理失败时，记录失败状态，比如 `sales_status = 2 同步失败`。

第二层是后台补偿任务定期扫描失败状态，再重新执行增加销量。如果服务在 `同步中` 状态崩溃，还会通过 `updated_at` 判断长时间卡住的记录，把它恢复成同步失败，再进入补偿流程。

### Q15：Outbox 本地消息表解决什么问题？

回答：

Outbox 解决的是“本地事务成功，但 MQ 发布失败”的一致性缺口。

如果订单状态已经从待支付改成已支付，但发布 `OrderPaidEvent` 时 MQ 不可用，那么后续增加销量事件可能丢失。

Outbox 的做法是：在本地事务里同时写业务数据和待发送消息：

```text
订单状态更新成功
  -> 同事务写 order_outbox_message
  -> 后台中继任务扫描 pending 消息
  -> 投递 RabbitMQ
  -> 投递成功后标记 sent
```

这样即使 MQ 当时不可用，消息也不会直接丢，后续中继任务可以继续投递。

### Q16：Outbox 和 MQ 是什么关系？

回答：

MQ 是消息通道，负责把事件从生产者传给消费者。

Outbox 是可靠发送模式，解决业务数据库事务和 MQ 发布之间不能原子提交的问题。

简单说：

```text
MQ 负责传消息；
Outbox 负责保证该发的消息不会因为发布失败直接丢掉。
```

### Q17：你了解 RabbitMQ 的哪些点？

回答要克制：

我主要掌握项目里用到的基础模型：producer、exchange、routing key、queue、consumer、ack，以及业务侧幂等处理。这个项目没有把重点放在 RabbitMQ 高级特性上，比如复杂路由、集群、高可用、死信队列治理等。我更关注的是业务事件异步化后，如何用状态字段、幂等和补偿保证最终一致性。

如果面试官继续追高级特性，可以承认边界：

RabbitMQ 高可用和复杂队列治理我没有深入展开，当前项目主要是基础异步消息和业务一致性实践。

## 六、Redis 缓存高频问题

### Q18：商品详情缓存怎么设计？

回答：

用 Cache Aside 模式。

读取时先查 Redis：

```text
查 Redis goods:detail:{goods_id}
  -> 命中：直接返回
  -> 未命中：查 MySQL
  -> MySQL 有数据：写正常缓存
  -> MySQL 无数据：写空值缓存
```

更新或删除商品时，先更新 MySQL，再删除 Redis 缓存。下一次读请求回源 MySQL 并重建缓存。

### Q19：缓存穿透怎么处理？

回答：

缓存穿透是请求一个不存在的数据，Redis 查不到，MySQL 也查不到，每次都打到 MySQL。

项目里用空值缓存处理：MySQL 查不到商品时，也写一个短 TTL 的空值标记，比如 `__EMPTY__`。下次同样的不存在 ID 进来，Redis 命中空值缓存，直接返回不存在，不再打 MySQL。

空值缓存 TTL 要短，因为后续可能真的创建这个商品，不能被旧空值挡太久。

### Q20：缓存击穿怎么处理？

回答：

缓存击穿是热点 key 过期，大量请求同时回源 MySQL。

项目里做的是单实例本地互斥版本：

```text
第一次查 Redis miss
  -> 按 goods_id 加本地锁
  -> 拿到锁后第二次查 Redis
  -> 仍 miss 才查 MySQL
  -> 写回 Redis
```

第二次查 Redis 是为了避免前一个请求已经重建好缓存，后面的请求还继续打 MySQL。

边界也要说清楚：

这个本地锁只保护单个 goods-service 进程。多实例下应该升级为 Redis 分布式锁，使用 `SET NX EX`、唯一 token 和 Lua 原子释放锁。

### Q21：缓存雪崩怎么处理？

回答：

缓存雪崩是大量 key 同时过期，导致请求集中打到 MySQL。

项目里通过 TTL 随机化降低风险：

```text
正常缓存 TTL = 基础时间 + 随机抖动
空值缓存 TTL = 短基础时间 + 随机抖动
```

随机化不是为了精确控制缓存时间，而是为了错开大量 key 的过期时间。

### Q22：为什么更新商品后删除缓存，而不是更新缓存？

回答：

MySQL 是主数据源，Redis 是读缓存。

更新 MySQL 后删除缓存更简单，下一次读时回源 MySQL 重建缓存。直接双写缓存会增加一致性风险，比如 MySQL 更新成功但缓存更新失败，或者并发读写导致缓存写回旧值。

删除缓存失败时可以记录 warning，不阻断主流程，依赖 TTL、补偿或排障恢复。

## 七、Elasticsearch / CDC 高频问题

### Q23：为什么商品搜索不用 MySQL LIKE？

回答：

MySQL LIKE 可以做简单模糊查询，但电商搜索通常需要多字段检索、中文分词、相关性排序、高亮、过滤和排序组合。Elasticsearch 更适合做搜索读模型。

在这个项目里，MySQL 仍然是商品主库，ES 只是搜索读模型。商品变更通过 CDC 同步到 ES。

### Q24：MySQL 数据怎么同步到 ES？

回答：

search-service 监听 MySQL Binlog。商品表发生 insert、update、delete 时，Binlog 里会有对应事件。

处理方式：

```text
insert -> 解析新行 -> upsert 到 ES
update -> RowsEvent 里 oldRow/newRow 成对出现 -> 取 newRow 更新 ES
delete -> 解析旧行 id -> 删除 ES 文档
```

这样商品数据以 MySQL 为准，ES 作为搜索索引跟随更新。

### Q25：MQ、Outbox、CDC 的边界是什么？

回答：

三者解决的问题不同。

```text
MQ：表达业务事件，驱动业务动作，比如订单已支付后增加销量。
Outbox：保证本地事务成功后，业务事件不会因为 MQ 发布失败直接丢失。
CDC：监听数据库变更，同步搜索、报表、审计等读模型。
```

判断标准：

```text
要驱动业务动作，用 MQ。
要保证事务和消息一致，用 Outbox。
要把数据库变化同步到搜索读模型，用 CDC。
```

### Q26：ES 里 `match` 和 `term` 有什么区别？

回答：

`match` 会走分词，适合商品名、详情、标签这类全文搜索。

`term` 不分词，适合品牌、状态、类型这类精确匹配字段。

项目里关键词搜索用 multi-match 查商品名、品牌、标签、详情。品牌过滤、价格区间过滤走 filter，不参与相关性打分。

### Q27：为什么 tags 要加 text 子字段？

回答：

原来的 `tags` 如果是 keyword，会把整个字符串当成一个值，不适合中文关键词搜索。

所以给 `tags` 加 multi-field：

```text
tags       保留 keyword，用于精确过滤或聚合
tags.text  使用 text + IK 分词，用于全文搜索
```

这样一个字段同时支持精确匹配和中文分词搜索。

## 八、互动模块高频问题

### Q28：评论怎么建模？

回答：

评论模型用了三个核心字段：

```text
parent_id      直接回复哪条评论
root_id        属于哪条一级评论
reply_user_id  回复的是哪个用户
```

`parent_id` 解决直接父评论，`root_id` 解决按一级评论聚合回复，`reply_user_id` 方便前端展示“回复 @谁”。

这些字段不能信前端传值，必须由服务端根据 `parent_id` 查询父评论后计算，避免跨商品回复或伪造回复关系。

### Q29：评论列表怎么避免 N+1 查询？

回答：

分页只查一级评论：

```text
object_id + type + parent_id = 0
```

拿到当前页一级评论 ID 后，用一次批量查询查回复：

```text
root_id IN (...)
```

然后在 Go 里按 `root_id` 分组，把 replies 挂回对应一级评论。

这样不会查 10 条一级评论再循环查 10 次回复。

### Q30：点赞怎么防重复？

回答：

点赞记录表用唯一索引兜底：

```text
user_id + type + object_id
```

同一个用户对同一个对象重复点赞时，插入会触发唯一索引冲突。业务上把这个冲突当成幂等成功，而不是返回错误。

取消点赞也要幂等：如果目标状态是“未点赞”，重复取消时没删到记录也应该返回成功。

### Q31：点赞计数怎么保证一致？

回答：

点赞记录和冗余计数要在一个本地事务里完成。

点赞时：

```text
校验评论存在
插入 praise_info
comment_info.like_count + 1
```

取消点赞时：

```text
删除 praise_info
如果确实删除到记录，comment_info.like_count - 1
```

重复点赞不会加计数，重复取消不会扣计数。扣减时带 `like_count > 0` 条件，避免异常情况下计数变成负数。

### Q32：为什么点赞列表必须按 user_id 查？

回答：

点赞和收藏属于用户行为列表，不能只按 type 查。如果不带当前用户 ID，会把全站用户的点赞 / 收藏记录暴露给当前用户。

所以 H5 网关必须从登录态获取当前用户，传给 interaction-service，查询时按 `type + user_id` 过滤。

## 九、秒杀 / 高并发口径

### Q33：秒杀怎么防超卖？

回答：

普通下单可以用 MySQL 条件更新防超卖，但秒杀是热点商品，所有请求打到同一行 MySQL 会有严重锁竞争。

秒杀链路通常会把活动库存预热到 Redis，用 Lua 脚本把判断库存、防重复参与、扣减库存放在一个原子操作里。

Redis 扣减成功只表示用户拿到了秒杀资格，后续通过 MQ 或本地消息表异步创建订单。订单和库存最终仍然要落到 MySQL，并通过失败补偿、超时取消和库存校准兜底。

### Q34：为什么 Redis Lua 能防超卖？

回答：

因为 Lua 脚本在 Redis 里原子执行。判断库存、判断用户是否已参与、扣库存、记录用户参与状态可以放在同一个脚本里，执行过程中不会被其他命令插入。

这样不会出现两个请求同时看到库存大于 0，然后都扣减成功的问题。

### Q35：Redis 扣库存成功，但订单落库失败怎么办？

回答：

要补偿。

Redis 扣库存成功只是预扣资格。如果后续创建订单失败，需要把 Redis 库存回补，并清理或更新用户秒杀结果状态。

更稳的做法是记录秒杀请求结果，比如：

```text
processing
success
failed
```

用户前端轮询结果，系统后台对失败任务做补偿。

### Q36：怎么防止一个用户重复秒杀？

回答：

用 `user_id + activity_id + goods_id` 作为唯一参与维度。

在 Redis Lua 脚本里先判断用户是否已经参与，如果已经参与直接返回重复请求；如果没参与且库存大于 0，再扣库存并记录用户参与标记。

MySQL 订单表或秒杀订单表也应该有唯一索引兜底，防止极端情况下重复落单。

### Q37：简历里秒杀怎么讲才稳？

回答：

口径要统一：

```text
单体电商项目里完整实践过秒杀链路；
微服务项目里把秒杀作为高并发交易链路的扩展设计，复用 Redis Lua、异步下单、幂等和补偿思路。
```

如果面试官追问代码，可以落到单体项目里的实现；如果问微服务项目怎么扩展，就按 Redis 预扣库存、MQ / Outbox 异步创建订单、MySQL 最终落库、失败补偿这套讲。

不要主动说“这个微服务项目里秒杀没完全做完”。只需要讲清楚你掌握的实现方案和已有实践边界。

### Q38：Docker Compose 编排了哪些服务？


H5 网关对外提供前台接口，manage-service 托管 GFast 管理后台静态资源和管理接口，业务服务通过 gRPC 协作。

### Q39：你怎么验证部署成功？

回答：

我做了几类验证：

1. Compose 配置校验：`docker compose -f docker-compose.prod.yml config --quiet`。
2. 镜像构建：`docker compose -f docker-compose.prod.yml build`。
3. 服务健康状态：`docker compose ps`，核心服务为 healthy。
4. HTTP 冒烟：管理后台首页、H5 OpenAPI、搜索 Swagger 返回 200。
5. 业务冒烟：商品列表、搜索、轮播图、评论、购物车、注册用户。
6. MQ 冒烟：注册用户后 RabbitMQ 投递消息到 goods-service，消费者成功发放优惠券。
7. 数据库验证：MySQL 中能查到新增用户和发券记录。

### Q40：部署时遇到过什么真实问题？

回答：

可以讲两个问题。

第一个是 RabbitMQ 消费幂等里，如果消息没有 `MessageId`，原来代码用 `msg.Body[:32]` 生成兜底 ID。注册消息体比较短时会触发切片越界 panic。修复方式是改成对完整消息体做 SHA256，作为兜底消息 ID。

第二个是 goods-service 生产配置里只有 `redis.goods`，但通用幂等组件使用的是默认 Redis 客户端 `g.Redis()`，需要 `redis.default`。消费者启动后处理消息时找不到 Redis 配置。修复方式是在 goods 生产配置里补齐 `redis.default`。

这两个问题都说明部署验证不能只看容器启动，要真的打业务链路和看日志。

## 十一、危险问题的防守口径

### Q41：你 RabbitMQ 熟吗？

稳妥回答：

我主要熟悉项目里用到的基础异步消息模型，包括 exchange、routing key、queue、consumer、ack，以及业务侧幂等和补偿。更偏 RabbitMQ 集群、高可用、死信队列治理这些生产级高级能力，我目前没有深入做。

然后立刻把话题拉回项目：

我在这个项目里更关注的是支付成功后副作用异步化，以及消息可能重复、消费失败、发布失败时，业务状态怎么兜底。

### Q42：Kafka 熟吗？

回答：

Kafka 不是这个项目的主链路，我没有在简历里作为重点写。消息队列这块我当前主要用 RabbitMQ 做基础异步消息实践，重点理解业务事件、消费幂等和最终一致性。Kafka 的分区、消费组、offset、顺序消费这些我了解概念，但没有作为这个项目的主要实践点。

### Q43：你这个项目是真微服务吗，还是只是拆目录？

回答：

是真正按服务进程拆分的。H5 网关、用户、商品、订单、互动、搜索等服务都是独立服务，通过 gRPC 通信，并通过 etcd 做服务注册发现。Docker Compose 启动时也是多个容器和中间件一起运行。

同时我也会承认边界：

它还不是完整生产级微服务治理，比如没有做 K8s、完整监控告警、链路追踪和灰度发布。当前重点是服务拆分、接口协作、交易一致性和本地生产形态部署。

### Q44：为什么项目里用了这么多技术，会不会只是堆技术？

回答：

每个技术点对应一个明确问题。

```text
gRPC / etcd：解决服务拆分后的调用和服务发现。
RabbitMQ：解决支付成功后副作用异步执行。
Outbox：解决本地事务成功但消息发布失败。
Redis：解决商品详情读性能、热点缓存和分布式锁辅助。
Elasticsearch：解决商品多字段搜索和中文分词。
Docker Compose：解决多服务和中间件的一键启动验证。
```

所以不是为了堆技术，而是围绕电商链路里的具体问题逐步引入。

### Q45：这个项目最大的收获是什么？

回答：

最大收获是理解了服务端开发不能只看单个接口 CRUD，而要看完整业务链路。

比如订单支付成功后，主状态、库存、销量、消息、补偿任务、重复消费、部署日志都要一起考虑。一个接口返回成功，不代表整个业务链路可靠。这个项目让我把接口开发、数据库状态、缓存、消息、搜索和部署验证串成了完整闭环。

## 十三、最后的讲法原则

1. 先讲业务问题，再讲技术方案。
2. RabbitMQ 只讲基础消息、幂等、补偿，不主动展开 Kafka 式高级问题。
3. 秒杀讲“单体已实践 + 微服务可扩展口径”，不要主动暴露实现边界。
4. ES 讲搜索读模型，不要吹成搜索平台。
5. Redis 讲 Cache Aside、穿透、击穿、雪崩和一致性边界。
6. 部署讲 Docker Compose 本地生产形态，不要说成云原生生产部署。
7. 每个技术点都要能回答“它解决了什么业务问题”。
