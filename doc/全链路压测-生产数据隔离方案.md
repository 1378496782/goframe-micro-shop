# 全链路压测生产数据隔离方案

## 1. 项目架构概述

根据分析，本项目是基于GoFrame框架的微服务架构，主要包含以下组件：

- **网关层**：gateway-h5、gateway-admin、gateway-resource
- **微服务层**：order、goods、user、banner、interaction、search
- **基础设施**：MySQL、Redis、RabbitMQ、Elasticsearch、etcd
- **通信方式**：gRPC (通过pbentity和v1包)
- **外部依赖**：微信支付等第三方服务

## 2. 全链路压测数据隔离设计

### 2.1 请求标记机制

**方案设计：**

1. **压测标识头**：在所有HTTP/HTTPS请求中添加专用压测标识头
   ```
   X-Pressure-Test: true
   ```

2. **gRPC请求标记**：在所有gRPC请求中添加压测元数据
   ```go
   metadata.Pairs("pressure-test", "true")
   ```

3. **全局上下文传播**：通过context在各服务间传递压测标记
   ```go
   ctx = context.WithValue(ctx, "pressure-test", true)
   ```

4. **流量染色机制**：在所有日志中添加压测标识，便于问题排查
   ```go
   g.Log().WithCtx(ctx).WithFields(g.Map{
       "pressure-test": ctx.Value("pressure-test"),
   }).Info(ctx, "操作日志")
   ```

### 2.2 数据库隔离方案

**方案设计：**

1. **数据库前缀隔离**：为压测数据添加特定前缀标识
   ```go
   // 根据压测标记生成不同的数据标识
   func GenerateTestDataPrefix(ctx context.Context) string {
       isPressureTest := false
       if val := ctx.Value("pressure-test"); val != nil {
           isPressureTest = val.(bool)
       }
       if isPressureTest {
           return "TEST_"
       }
       return ""
   }
   ```

2. **表前缀方案**：创建压测专用表（适用于有条件的场景）
   - 生产表：`order_info`
   - 压测表：`pt_order_info`

3. **数据标识方案**（推荐）：在表中添加压测标记字段
   ```sql
   ALTER TABLE order_info ADD COLUMN is_pressure_test tinyint(1) DEFAULT 0 COMMENT '是否为压测数据';
   ```

4. **事务隔离**：确保压测数据操作不会影响生产数据
   ```go
   // 在DAO层添加压测数据筛选条件
   func (dao *OrderInfo) GetModel(ctx context.Context) *g.DB {
       model := dao.model
       if isPressureTest, ok := ctx.Value("pressure-test").(bool); ok && isPressureTest {
           model = model.Where("is_pressure_test = ?", 1)
       } else {
           model = model.Where("is_pressure_test = ?", 0)
       }
       return model
   }
   ```

### 2.3 缓存隔离方案

**方案设计：**

1. **缓存键前缀**：在所有缓存键中添加压测标识前缀
   ```go
   func GetCacheKey(ctx context.Context, key string) string {
       if isPressureTest, ok := ctx.Value("pressure-test").(bool); ok && isPressureTest {
           return "pt:" + key
       }
       return key
   }
   ```

2. **Redis数据库隔离**：使用不同的Redis数据库编号
   ```go
   func GetRedisDB(ctx context.Context) int {
       if isPressureTest, ok := ctx.Value("pressure-test").(bool); ok && isPressureTest {
           return 1  // 压测专用数据库
       }
       return 0  // 生产数据库
   }
   ```

3. **缓存过期策略**：为压测数据设置较短的过期时间
   ```go
   func SetCache(ctx context.Context, key string, value interface{}, duration time.Duration) error {
       cacheKey := GetCacheKey(ctx, key)
       if isPressureTest, ok := ctx.Value("pressure-test").(bool); ok && isPressureTest {
           // 压测数据设置较短过期时间
           duration = time.Minute * 5
       }
       return g.Redis().Set(ctx, cacheKey, value, duration)
   }
   ```

## 3. 实现步骤

1. **中间件实现**：创建全局压测标记中间件
2. **DAO层改造**：在数据访问层添加压测数据隔离逻辑
3. **服务层改造**：确保服务间调用正确传递压测标记
4. **外部接口封装**：实现压测环境的接口隔离
5. **监控适配**：确保监控系统能识别并区分压测流量

## 4. 验证机制

1. **日志验证**：通过日志确认压测标记正确传播
2. **数据验证**：确认压测数据不会与生产数据交互
3. **性能验证**：确认隔离措施不会显著影响性能
4. **故障恢复**：确认出现问题时可以快速隔离和恢复

## 5. 风险控制

1. **强制隔离检查**：在关键操作前强制检查隔离标记
2. **数据清理机制**：定期清理压测数据
3. **监控告警**：对跨域访问进行监控和告警
4. **权限控制**：严格控制压测功能的使用权限

此方案通过多层次的隔离机制，确保全链路压测过程中不会影响生产环境的正常运行，同时保持压测数据的真实性和有效性。
