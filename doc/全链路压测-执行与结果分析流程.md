# 全链路压测执行与结果分析流程

## 1. 文档目的

本文档定义了shop-goframe微服务项目全链路压测的标准执行流程和结果分析方法，旨在通过系统化的压测过程，全面评估系统在不同负载下的性能表现，识别潜在瓶颈，并为系统优化提供数据支持。

## 2. 压测前准备工作

### 2.1 压测目标确定

| 压测类型 | 目标说明 | 适用场景 |
|---------|---------|----------|
| 基准测试 | 评估系统在正常负载下的性能基准 | 新功能上线前、系统调优前后对比 |
| 负载测试 | 验证系统在预期负载下的性能表现 | 业务高峰期前评估 |
| 容量测试 | 确定系统的最大承载能力 | 系统扩容规划、架构设计验证 |
| 稳定性测试 | 评估系统在长时间运行下的稳定性 | 重要业务系统验收 |
| 异常测试 | 验证系统在异常情况下的容错能力 | 高可用架构验证 |

### 2.2 环境准备检查清单

```
□ 确认压测环境与生产环境配置一致性
```

## 4. 压测执行流程

### 4.1 压测执行步骤

#### 4.1.1 预热阶段

1. **目的**：让系统达到稳定状态，避免冷启动影响
2. **执行方式**：
   - 以目标负载的30%开始
   - 持续10分钟
   - 监控系统各项指标，确保正常

```bash
# 使用Locust命令行进行预热
locust -f locustfile.py --headless -u 300 -r 50 -t 10m --host http://target-service --csv=warmup
```

#### 4.1.2 逐步加压阶段

1. **目的**：渐进式增加负载，找到系统瓶颈
2. **执行方式**：
   - 从预热负载开始，每次增加20%负载
   - 每个梯度持续15分钟
   - 记录每个梯度的性能数据

**压测梯度表**：

| 梯度 | 并发用户数 | 持续时间 | 增长率 | 预期目标 |
|------|------------|---------|--------|----------|
| 1 | 300 | 15分钟 | 0% | 系统稳定运行 |
| 2 | 500 | 15分钟 | 67% | 响应时间正常 |
| 3 | 800 | 15分钟 | 60% | 无错误产生 |
| 4 | 1200 | 15分钟 | 50% | 资源使用率<70% |
| 5 | 1800 | 15分钟 | 50% | 系统仍可响应 |

#### 4.1.3 稳定运行阶段

1. **目的**：验证系统在目标负载下的稳定性
2. **执行方式**：
   - 维持目标并发用户数
   - 持续60分钟
   - 密切监控系统各项指标

```bash
# 稳定运行阶段的Locust命令
locust -f locustfile.py --headless -u 1000 -r 100 -t 60m --host http://target-service --csv=stable
```

#### 4.1.4 极限测试阶段

1. **目的**：确定系统的极限承载能力
2. **执行方式**：
   - 快速增加负载直到系统出现性能下降
   - 记录系统的最大RPS和临界并发数
   - 观察系统崩溃时的表现

### 4.2 压测过程监控

#### 4.2.1 实时监控要点

- **性能指标**：
  - 响应时间（平均值、95%、99%分位数）
  - 请求成功率
  - 吞吐量（RPS）
  
- **资源指标**：
  - CPU使用率
  - 内存使用率
  - 磁盘I/O
  - 网络吞吐量
  
- **业务指标**：
  - 订单成功率
  - 支付成功率
  - 退款处理时间

#### 4.2.2 紧急停止条件

当出现以下情况时，应立即停止压测：

1. 错误率超过5%
2. 平均响应时间超过3秒
3. CPU使用率持续超过95%
4. 内存使用率持续超过90%
5. 出现数据异常或污染生产数据
6. 系统出现崩溃或服务不可用

## 6. 结果分析方法

### 6.1 性能指标分析

#### 6.1.1 响应时间分析

**分析要点**：
- 识别响应时间异常的接口
- 分析响应时间分布情况
- 确定响应时间是否满足业务SLA

**示例分析**：

```
API: /api/order/create
- 平均响应时间: 320ms (目标: <500ms)
- 95%响应时间: 780ms (目标: <1s)
- 99%响应时间: 1200ms (目标: <1.5s)
结论: 满足性能要求，但接近95%阈值，建议优化
```

#### 6.1.2 吞吐量分析

**分析要点**：
- 计算系统的最大吞吐量
- 分析吞吐量随并发用户数的变化趋势
- 识别吞吐量瓶颈

**瓶颈识别方法**：

1. **线性增长阶段**：吞吐量与并发数成正比，系统未达到瓶颈
2. **缓慢增长阶段**：吞吐量增长放缓，系统开始出现瓶颈
3. **平稳阶段**：吞吐量不再增长，系统达到性能上限
4. **下降阶段**：吞吐量下降，系统出现严重问题

### 6.2 资源使用分析

#### 6.2.1 CPU使用分析

- **CPU饱和**：单个核心利用率达到100%，可能存在计算密集型操作
- **CPU不均衡**：部分核心负载高，部分低，可能存在线程池配置问题
- **CPU波动大**：CPU使用率不稳定，可能存在GC频繁或锁竞争

#### 6.2.2 内存使用分析

- **内存泄漏**：内存持续增长不释放
- **内存占用过高**：可能存在对象过大或缓存未限制
- **频繁GC**：导致系统停顿，影响响应时间

#### 6.2.3 数据库性能分析

**慢查询分析**：
```sql
-- 查看MySQL慢查询
SELECT id, user_id, db, command, time, state, info 
FROM information_schema.processlist 
WHERE time > 10 
ORDER BY time DESC;
```

**连接池分析**：
- 连接数使用率是否超过80%
- 是否存在连接泄漏
- 连接获取时间是否过长

### 6.3 错误分析

**错误分类**：

1. **超时错误**：响应时间超过阈值
2. **服务器错误**：5xx响应码
3. **业务错误**：4xx响应码
4. **系统异常**：异常堆栈或崩溃

**错误根因分析方法**：

1. **日志分析**：检查错误日志中的堆栈信息
2. **关联分析**：将错误与系统资源使用关联
3. **代码审查**：重点检查高频错误的代码路径
4. **压力复现**：在测试环境复现并调试

## 8. 压测经验总结

### 8.1 常见问题与解决方案

| 问题类型 | 症状 | 可能原因 | 解决方案 |
|---------|------|---------|----------|
| 响应时间过长 | 95%响应时间超过阈值 | 数据库查询慢、锁竞争、资源不足 | 优化SQL、减少锁范围、增加资源 |
| 错误率上升 | 请求失败率超过预期 | 资源耗尽、超时设置不合理、代码bug | 增加超时时间、修复代码、增加资源 |
| 内存泄漏 | 内存持续增长不释放 | 对象未回收、缓存未清理、连接泄漏 | 检查对象生命周期、清理缓存、关闭连接 |
| 吞吐量不达标 | RPS低于目标值 | 串行操作多、IO阻塞、算法效率低 | 并行处理、异步操作、优化算法 |

### 8.2 最佳实践建议

1. **压测前置条件**：
   - 确保代码质量良好，通过单元测试和集成测试
   - 确保监控系统覆盖所有关键指标
   - 确保数据隔离机制有效

2. **压测过程控制**：
   - 采用渐进式加压策略
   - 密切监控系统各项指标
   - 准备应急预案，出现问题及时停止

3. **结果分析要点**：
   - 关注瓶颈点和拐点
   - 结合业务指标和技术指标
   - 进行多维度对比分析

4. **持续改进**：
   - 建立性能基线
   - 定期进行压测
   - 跟踪性能变化趋势

### 9.2 性能指标计算公式

- **平均响应时间**：总响应时间 / 请求总数
- **95%响应时间**：将所有请求的响应时间排序，第95%位置的值
- **吞吐量（RPS）**：请求总数 / 总时间（秒）
- **错误率**：失败请求数 / 总请求数 × 100%
- **并发用户数**：同时在线的虚拟用户数
