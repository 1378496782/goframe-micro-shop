# 互动模块学习记录

互动模块后面要单独实战一轮。

这块不是可有可无的边角功能。多级评论、点赞幂等、计数一致性是后端面试高频题，也很适合练业务建模能力。

## 学习目标

围绕商品评论 / 回复 / 点赞，学清楚：

```text
评论怎么建模
回复怎么查询
点赞怎么防重复
点赞计数怎么维护
高并发点赞怎么抗压
```

## 当前进度

### 已完成：点赞 / 取消点赞基础幂等

这一小步完成的是互动模块里最基础、也最容易踩坑的用户行为表改造。

当前已经跑通：

- H5 网关创建点赞时，不再相信前端传 `user_id`，而是从登录态 `CtxUserId` 获取当前用户。
- `praise_info` gRPC 请求补充 `user_id`，由网关传给 `interaction-service`。
- 创建点赞时依赖数据库唯一索引 `user_id + type + object_id` 防重复。
- 重复点赞触发 MySQL `Duplicate entry / Error 1062` 时，按幂等成功处理，不再把错误返回给前端。
- 取消点赞时按 `id + user_id` 删除，避免用户 A 删除用户 B 的点赞。
- 重复取消点赞按幂等成功处理：目标状态是“没有点赞记录”，没删到数据也不应该报业务错误。

这一小步的核心结论：

```text
点赞不是简单插入一条记录。
必须同时考虑：
1. 用户身份只能来自 token / 上下文，不能来自前端参数
2. 唯一索引是防重复的最终兜底
3. 重复点赞和重复取消都应该幂等成功
4. 删除用户行为记录时必须带 user_id，避免越权删除
```

当前暂时没有做：

- 点赞数量 `like_count` 冗余维护
- Redis 点赞计数

这些放到后续小步继续做。

### 已完成：点赞 / 收藏列表按当前用户隔离查询

这一小步解决的是用户行为列表的查询边界。

当前已经跑通：

- 收藏列表原本已经从 H5 网关读取 `CtxUserId`，并通过 gRPC 传给 `interaction-service`。
- 收藏列表在 interaction-service 查询时已经带上 `type + user_id` 条件。
- 点赞列表补齐了同样的用户隔离链路：H5 网关读取当前登录用户，设置到 `PraiseInfoGetListReq.UserId`。
- `praise_info` gRPC 查询请求补充 `user_id` 字段。
- 点赞列表的总数查询和分页列表查询都增加 `type + user_id` 条件，避免把其他用户的点赞记录查出来。

这一小步的核心结论：

```text
用户行为列表不能只按 type 查。
必须按当前登录用户 user_id 过滤，否则会把全站用户的点赞 / 收藏记录暴露给当前用户。
```

当前暂时没有做：

- 点赞数量 `like_count` 冗余维护
- Redis 点赞计数

### 已完成：评论最小可用版

这一小步先没有大改评论表，而是基于现有 `comment_info.parent_id` 做了评论链路的最小可用版本。

当前已经跑通：

- 创建评论时，H5 网关从登录态 `CtxUserId` 获取当前用户，并传给 `interaction-service`。
- 评论列表支持按 `object_id + type + parent_id` 查询。
- 当 `parent_id = 0` 时，列表只返回一级评论，不会把二级回复混进来。
- 评论列表排除 `deleted_at` 非空的数据。
- 删除评论时按 `id + user_id` 删除，避免用户 A 删除用户 B 的评论。

这一小步的核心结论：

```text
评论列表不能全表分页。
至少要按 object_id + type 限定业务对象；
如果当前只展示一级评论，还必须固定 parent_id = 0；
删除评论必须带 user_id 做用户隔离。
```

当前暂时没有做：

- `root_id / reply_user_id` 字段升级
- 评论数量 / 回复数量冗余维护
- 评论点赞计数

### 已完成：一级评论下的回复列表

这一小步把评论列表从单层列表升级成两层结构。

当前已经跑通：

- 一级评论仍然按 `object_id + type + parent_id = 0` 分页查询。
- 查询到当前页一级评论后，收集这一页的评论 ID。
- 使用 `parent_id IN (...)` 一次性批量查询这些一级评论下面的回复。
- 在 Go 里按 `parent_id` 分组，再把 `replies` 挂回对应的一级评论。
- H5 响应结构增加 `replies` 字段，前端可以直接渲染两层评论。

这一小步的核心结论：

```text
评论列表不要查 10 条一级评论，再循环查 10 次回复。
正确做法是：
1. 分页查一级评论
2. 收集一级评论 id
3. 批量查 parent_id IN (...)
4. 内存分组后挂回 replies
```

当前暂时没有做：

- 评论数量 / 回复数量冗余维护
- 评论点赞计数

### 已完成：评论模型升级 root_id / reply_user_id

这一小步把评论模型从“只能知道直接父评论”升级成“能知道所属一级评论和被回复用户”。

当前已经跑通：

- `comment_info` 表新增 `root_id` 和 `reply_user_id` 字段。
- DAO / Entity / DO 已根据本地表结构重新生成。
- `pbentity.CommentInfo` 和 H5 `CommentInfoItem` 都返回 `root_id / reply_user_id`。
- 创建评论时，`root_id / reply_user_id` 不由前端传入，而是在 `interaction-service` 里根据 `parent_id` 查询父评论后计算。
- 查询父评论时会校验 `object_id + type + deleted_at IS NULL`，避免跨对象回复或回复已删除评论。
- 初始化 SQL 已同步新增字段和索引，避免重建数据库后代码和表结构不一致。

这一小步的核心结论：

```text
parent_id 解决“我直接回复谁”；
root_id 解决“我属于哪条一级评论下面”；
reply_user_id 解决“前端展示回复 @谁”。

这些字段不能信前端传值，必须由服务端根据父评论计算。
```

当前暂时没有做：

- 评论数量 / 回复数量冗余维护
- 评论点赞计数

### 已完成：回复列表查询升级为 root_id IN

这一小步把一级评论下的回复查询从“只查直接子回复”升级成“查整组后代回复”。

当前已经跑通：

- 一级评论分页查询仍然固定 `parent_id = 0`，不会影响顶层评论分页。
- 查询到当前页一级评论后，收集一级评论 ID 作为 `rootIds`。
- 回复批量查询从 `parent_id IN (...)` 改为 `root_id IN (...)`。
- 回复分组也从按 `parent_id` 分组改为按 `root_id` 分组。
- 最终仍然把回复平铺挂到一级评论的 `replies` 下，先不做无限层级树形组装。

这一小步的核心结论：

```text
parent_id IN 只能查到一级评论的直接子回复；
root_id IN 可以查到某条一级评论下面的所有后代回复。

有了 root_id，评论列表就能支持“回复别人的回复”，同时仍然避免 N+1 查询。
```

当前暂时没有做：

- 评论点赞计数

### 已完成：评论列表返回 like_count / reply_count

这一小步把评论列表的展示数据从“只有评论内容和回复列表”升级成“带基础计数”。

当前已经跑通：

- `comment_info` 表新增 `like_count` 和 `reply_count` 字段，默认值为 0。
- 本地历史数据已按 `root_id` 回填一级评论的 `reply_count`。
- `pbentity.CommentInfo` 和 H5 `CommentInfoItem` 都返回 `like_count / reply_count`。
- 评论列表转换链路已经打通：DB -> Entity -> pbentity -> H5 response。
- 创建回复时，如果 `parent_id > 0`，会给所属一级评论的 `reply_count + 1`。
- 初始化 SQL 已同步新增字段，避免重建数据库后缺列。

这一小步的核心结论：

```text
reply_count 是一级评论下回复总数，创建回复时维护在 root 评论上；
like_count 先只作为返回字段打通，真正的点赞/取消点赞计数维护放下一小步。
```

当前暂时没有做：

- 点赞计数的事务一致性和 Redis 计数方案

### 已完成：点赞 / 取消点赞维护 MySQL 冗余 like_count

这一小步把评论点赞从“只记录用户行为”升级成“用户行为记录 + 评论冗余计数”。

当前已经跑通：

- `praise_info.type = 2` 约定为评论 / 回复点赞。
- 点赞评论前会校验目标评论存在且未删除，避免给不存在的评论插点赞记录。
- 点赞成功插入 `praise_info` 后，才会对 `comment_info.like_count + 1`。
- 重复点赞命中唯一索引时按幂等成功处理，不会重复增加 `like_count`。
- 取消点赞按 `id + user_id + type + object_id` 删除，避免删错点赞记录。
- 只有真正删除到点赞记录时，才会对 `comment_info.like_count - 1`。
- 重复取消点赞 `RowsAffected = 0` 时按幂等成功处理，不会重复扣减。
- 扣减时带 `like_count > 0` 条件，避免异常情况下把计数扣成负数。

这一小步的核心结论：

```text
like_count 不能直接根据请求加减。
必须先看 praise_info 是否真的发生了状态变化：
1. 新增点赞记录成功，才 like_count + 1
2. 重复点赞命中唯一索引，不加计数
3. 删除点赞记录成功，才 like_count - 1
4. 重复取消没删到记录，不扣计数
```

当前暂时没有做：

- Redis 高并发点赞计数
- 定时校准 `praise_info` 和 `comment_info.like_count`

### 已完成：点赞计数事务一致性

这一小步把评论点赞计数从“两个独立数据库写操作”升级成“一个本地事务内完成”。

当前已经跑通：

- 评论 / 回复点赞时，目标评论存在性校验、`praise_info` 插入、`comment_info.like_count + 1` 都放在同一个事务里。
- 评论 / 回复取消点赞时，`praise_info` 删除和 `comment_info.like_count - 1` 放在同一个事务里。
- 事务内统一使用 `tx.Model(...)` 操作 `praise_info` 和 `comment_info`，避免混用非事务 DAO。
- 重复点赞命中唯一索引时仍然按幂等成功处理，不会增加 `like_count`。
- 重复取消点赞 `RowsAffected = 0` 时仍然按幂等成功处理，不会扣减 `like_count`。
- `like_count` 扣减继续保留 `like_count > 0` 条件，避免异常情况下出现负数。

这一小步的核心结论：

```text
用户行为记录和冗余计数必须同生共死。

点赞记录插入成功但 like_count 更新失败，会导致展示计数偏小；
点赞记录删除成功但 like_count 更新失败，会导致展示计数偏大。

所以 MySQL 版本的点赞计数，至少要把 praise_info 状态变化和 comment_info.like_count 更新放进同一个本地事务。
```

当前暂时没有做：

- Redis 异步落库方案
- 多实例下的分布式校准锁

### 已完成：Redis 点赞计数旁路缓存

这一小步把评论点赞计数从“只读 MySQL 冗余字段”升级成“Redis 缓存优先，MySQL 兜底”。

当前已经跑通：

- 新增互动模块 Redis 工具包 `interactionRedis`。
- 评论点赞计数 key 使用 `interaction:comment:like_count:{comment_id}`。
- 创建点赞真正新增记录后，才会对 Redis 计数 `INCR`。
- 重复点赞命中唯一索引时仍然幂等成功，不更新 Redis 计数。
- 取消点赞真正删除记录后，才会尝试扣减 Redis 计数。
- Redis key 不存在时取消点赞不做 `DECR`，避免把缺失 key 扣成 `-1`。
- 评论列表转换时优先读取 Redis 的 `like_count`，未命中时回退 MySQL 并写回 Redis。
- Redis key 使用 TTL，`Set` 使用 `SetEX`，`INCR/DECR` 后刷新过期时间。
- `interaction-service` 已补充 Redis adapter 和 `redis.default` 配置。

这一小步的核心结论：

```text
Redis 在当前阶段只是点赞计数缓存，不是权威数据。

所以：
1. MySQL 事务成功后再更新 Redis
2. Redis 更新失败不影响主业务
3. Redis miss 时用 MySQL like_count 回填
4. Redis key 必须设置 TTL，避免长期堆积历史计数
```

### 已完成：点赞计数定时校准

这一小步补上了 Redis / MySQL 冗余计数的兜底修复能力。

当前已经跑通：

- 新增 `CalibrateCommentLikeCount`，以 `praise_info` 真实点赞记录为准反算评论点赞数。
- 校准扫描使用 `id > lastId` 游标分页，而不是只 `Limit(N)` 扫前 N 条。
- 每批评论使用 `object_id IN (...) GROUP BY object_id` 聚合统计点赞数，避免逐条 `COUNT(*)`。
- 当真实点赞数和 `comment_info.like_count` 不一致时，修正 MySQL 冗余字段。
- 无论 MySQL 是否需要修正，都会把真实点赞数写回 Redis，并复用 Redis 工具里的 TTL。
- 新增后台任务，随 `interaction-service` 启动后立即跑一轮，之后按固定间隔校准。
- 后台任务使用进程内 `running` 标记防止同一实例内重入。

这一小步的核心结论：

```text
冗余计数一定要有校准口。

praise_info 是用户行为事实表；
comment_info.like_count 是查询性能冗余；
Redis like_count 是缓存。

当三者不一致时，以 praise_info 重新计算出来的结果为准。
```

当前暂时没有做：

- Redis 异步落库：当前点赞仍然同步写 MySQL，再更新 Redis
- 多实例分布式锁：当前校准任务只做单实例进程内防重入
- 校准任务管理端手动触发入口

### 高并发点赞升级方案

当前项目里的实现是稳妥版：

```text
请求 -> MySQL 写 praise_info + comment_info.like_count -> Redis 更新缓存
```

这个方案适合学习幂等、事务和缓存兜底，优点是准确、简单、容易排查；缺点是大量用户同时点赞时，请求主链路仍然会打到 MySQL。

如果同一条评论 / 商品被大量用户同时点赞，可以升级为：

```text
请求 -> Redis 原子判断是否首次点赞 -> Redis 更新计数 -> 异步落 MySQL -> 定时校准兜底
```

典型 Redis key：

```text
comment:like:users:{comment_id}      Set / Bitmap，记录哪些用户点过赞
comment:like:count:{comment_id}      String，记录实时点赞数
comment:like:dirty_set               Set，记录哪些评论的点赞计数发生过变化
```

#### Redis Set 版本

点赞流程：

```text
SADD comment:like:users:{comment_id} user_id

如果返回 1：
  说明是首次点赞
  INCR comment:like:count:{comment_id}
  SADD comment:like:dirty_set comment_id

如果返回 0：
  说明是重复点赞
  直接幂等成功，不重复加计数
```

取消点赞流程：

```text
SREM comment:like:users:{comment_id} user_id

如果返回 1：
  说明点赞状态真的发生变化
  DECR comment:like:count:{comment_id}
  SADD comment:like:dirty_set comment_id

如果返回 0：
  说明用户本来就没点赞
  直接幂等成功，不重复扣计数
```

`SADD/SREM/INCR/DECR` 都是 Redis 原子命令。更严谨时可以用 Lua 把“判断状态 + 改计数 + 标记 dirty”放进一个脚本里，保证一组动作整体原子。

#### Bitmap 版本

如果点赞用户非常多，Redis Set 会比较占内存。Bitmap 可以用一个 bit 表示一个用户是否点赞：

```text
comment:like:bitmap:{comment_id}
```

示例：

```text
GETBIT comment:like:bitmap:{comment_id} user_id
SETBIT comment:like:bitmap:{comment_id} user_id 1
BITCOUNT comment:like:bitmap:{comment_id}
```

优点：

```text
1 个用户理论上只占 1 bit，内存非常省；
适合用户 ID 连续、用户量很大的点赞状态判断。
```

缺点：

```text
如果 user_id 很大且稀疏，会因为 offset 过大浪费空间；
不适合分页展示“谁点了赞”；
不适合存点赞时间、设备、来源等明细字段。
```

所以 Bitmap 更适合“判断是否点赞 + 统计点赞数”，不适合替代 `praise_info` 这种用户行为明细表。

#### MQ 是否每次都发

高并发点赞不一定每次 `INCR` 都发一条 MQ。

如果“首次点赞一次 = 一条 MQ 消息”，在热点场景下 MQ 消息量会和点赞 QPS 一样高，容易造成：

```text
消息积压
消费延迟
磁盘压力上升
死信 / 过期策略被触发
```

更适合热点点赞的是 dirty set / 聚合落库：

```text
点赞请求只把 comment_id 放进 dirty_set；
同一条评论 1 万次点赞，dirty_set 里也只有一个 comment_id；
后台任务定期读取 dirty_set，批量读取 Redis 最新 count，批量更新 MySQL。
```

如果必须通过 MQ，可以发聚合消息，而不是单用户点赞消息：

```json
{
  "comment_id": 11,
  "like_delta": 328
}
```

但如果还要保存每个用户的点赞明细，仍然需要批量同步用户维度状态，不能只保存聚合计数。

这一块的核心结论：

```text
低中并发：MySQL 唯一索引 + 冗余 like_count，简单准确。
高并发：Redis 原子判重 + Redis 计数 + dirty set / 聚合异步落库。
超高并发热点：考虑 Bitmap / 分片 Set，避免单个 liked_users key 过大。
最终一致性：定时用 praise_info 反算，校准 Redis 和 comment_info.like_count。
```

下一步进入互动模块收尾：做一次评论 / 点赞 / 收藏整体 review，整理面试回答，并判断是否进入下一个大模块。

## 1. 评论表设计

先看现有 `interaction-service` 里的 `comment_info` 表，再判断它是否能支撑多级评论。

重点字段通常包括：

```text
id
user_id
goods_id / biz_id
content
parent_id
root_id
reply_user_id
like_count
reply_count
status
created_at
updated_at
```

核心理解：

- `parent_id` 表示当前评论直接回复的是哪条评论
- `root_id` 表示当前评论属于哪条一级评论下面
- 一级评论：`parent_id = 0`
- 二级回复：`parent_id = 被回复评论ID`，`root_id = 一级评论ID`

面试里要能讲清楚：

```text
为什么需要 root_id？
```

因为只靠 `parent_id` 查询一棵评论树，容易递归查询。有了 `root_id`，可以快速查某个一级评论下面的所有回复。

## 2. 评论列表查询

推荐先做两层展示，不急着支持无限层级完整展开。

典型接口：

```text
GET /frontend/comments?goods_id=51&page=1&size=10
```

返回结构：

```json
{
  "list": [
    {
      "id": 1,
      "content": "这个商品不错",
      "like_count": 10,
      "reply_count": 3,
      "replies": [
        {
          "id": 11,
          "content": "我也觉得",
          "reply_user_id": 2
        }
      ]
    }
  ],
  "total": 100
}
```

查询思路：

1. 分页查一级评论：`parent_id = 0`
2. 收集一级评论 ID
3. 批量查这些一级评论下的前 N 条回复：`root_id IN (...)`
4. 在 Go 里按 `root_id` 分组，挂回一级评论

要避免：

```text
查 10 条一级评论，然后循环查 10 次回复。
```

这就是典型 N+1 查询问题。

## 3. 点赞 / 取消点赞设计

点赞不是简单 `like_count + 1`。

需要一张点赞记录表，例如 `praise_info`：

```text
id
user_id
biz_type
biz_id
status
created_at
updated_at
```

`biz_type` 用来区分点赞对象：

```text
1 商品
2 评论
3 回复
```

必须加唯一约束：

```sql
UNIQUE KEY uk_user_biz (user_id, biz_type, biz_id)
```

这样才能防止同一个用户重复点赞同一个对象。

点赞接口要做到幂等：

```text
用户已经点赞，再点点赞：返回成功，不重复 +1
用户未点赞，点点赞：写点赞记录，like_count +1
```

取消点赞也要幂等：

```text
用户已经取消，再取消：返回成功，不重复 -1
用户已点赞，取消：更新点赞状态，like_count -1
```

## 4. 点赞计数维护

有三种方案需要理解。

方案 A：实时 `COUNT(*)`

```sql
SELECT COUNT(*) FROM praise_info WHERE biz_type = 2 AND biz_id = ? AND status = 1
```

优点：准确。

缺点：每次列表都 count，数据大了慢。

方案 B：评论表冗余 `like_count`

```sql
UPDATE comment_info
SET like_count = like_count + 1
WHERE id = ?
```

优点：列表查询快。

缺点：要处理重复点赞、并发、事务一致性。

方案 C：Redis 计数 + 异步落库

```text
Redis INCR comment:like_count:{comment_id}
后台定时同步到 MySQL
```

优点：抗热点能力强。

缺点：一致性更复杂，需要补偿和定时校准。

当前学习建议：

```text
先用 MySQL 唯一索引 + 冗余 like_count。
等点赞链路跑通后，再学习 Redis 计数优化。
```

## 5. 并发和一致性问题

面试常问：

```text
两个请求同时点赞同一条评论，会不会重复加计数？
```

回答思路：

1. `praise_info` 用唯一索引防重复记录
2. 插入成功才允许 `like_count + 1`
3. 如果插入失败是重复点赞，不加计数，返回成功
4. 计数更新使用数据库原子更新：`like_count = like_count + 1`

取消点赞也类似：

```text
只有从 已点赞 -> 已取消 的状态变更成功，才允许 like_count - 1。
```

可以用条件更新：

```sql
UPDATE praise_info
SET status = 0
WHERE user_id = ?
  AND biz_type = ?
  AND biz_id = ?
  AND status = 1
```

如果 `RowsAffected = 1`，再扣减计数。

## 6. 删除评论

评论删除不要一上来物理删除。

建议先学软删除：

```text
status = 0 正常
status = 1 已删除
```

展示时可以显示：

```text
该评论已删除
```

而不是直接删掉整棵回复树。

这样能避免：

- 子回复找不到父评论
- 评论计数错乱
- 审计和风控无法追溯

## 8. 面试回答模板

如果面试问：

```text
多级评论和点赞计数你怎么设计？
```

可以这样答：

```text
评论我会用 parent_id + root_id 建模。
parent_id 表示直接回复谁，root_id 表示属于哪条一级评论。
列表查询时先分页查一级评论，再批量查这些一级评论下的前几条回复，避免 N+1 查询。

```
