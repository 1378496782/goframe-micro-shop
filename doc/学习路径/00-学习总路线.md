# 学习总路线

这份文档是后续学习的主线锚点。

每次不知道下一步做什么时，先看这里，再结合当前代码状态决定下一轮的小任务。

## 当前进度

已经完成或基本跑通：

- 商品列表：分页、搜索、分类、价格区间、排序、库存筛选
- 商品详情：Redis 缓存、空缓存、缓存失效
- 购物车：加购、重复商品合并、更新数量、列表汇总
- 订单预览：从购物车选择商品并计算总价
- 从购物车创建订单：创建订单、保存商品快照、扣库存、删除购物车
- 取消订单：状态流转、恢复库存、失败补偿
- 支付成功加销量：支付回调后增加商品销量
- 销量补偿：`sales_status=2` 的订单可手动触发补偿

## 当前下一步

```text
补偿任务健壮性：
处理 sales_status = 3 同步中 卡住的问题。
```

这一轮要解决的问题：

```text
补偿任务抢占订单后，已经把 sales_status 从 2 改成 3。
如果服务在调用 goods-service 前后突然挂了，这个订单可能一直停在同步中。
```

目标：

```text
能识别超时卡住的同步中订单，并让它重新进入可补偿状态。
```

## 主线图

```text
同步业务链路
  -> 状态流转
  -> 补偿机制
  -> 后台任务

  -> 本地消息表 / Outbox

  -> Redis 高并发缓存
  -> 互动模块实战

```

## 后续学习阶段

### 1. 补偿任务健壮性

- `sales_status=3 同步中` 为什么会卡住
- 如何用 `updated_at` 判断补偿任务是否超时
- 如何把超时同步中的订单重新拉回同步失败
- 如何避免多个补偿任务重复处理同一单

目标：知道补偿任务不只是重试失败，还要处理“处理中但中断”的状态。

### 2. 订单状态机整理

- 待支付、已支付、已取消、已发货、已完成怎么流转
- 为什么状态更新要带 `FromStatus`
- 支付回调为什么要幂等
- 取消订单、超时取消、支付成功之间有没有冲突

目标：能画出订单状态机，并知道每条状态边由哪个业务动作触发。

### 3. 订单超时未支付自动取消

- 定时扫描待支付订单
- 超时后自动取消订单
- 自动取消后恢复库存
- 多实例部署下如何避免重复取消

目标：把状态流转、库存恢复、补偿思想串起来。

### 4. 补偿任务入口升级

当前补偿是手动 API 触发，后续要升级为后台任务。

- 后台任务放在哪里启动
- 日志怎么观察
- 多实例部署时如何避免重复执行
- 手动触发入口是否保留给管理端排障

目标：理解后台任务和普通 HTTP 接口的职责区别。

### 5. MQ / RabbitMQ 最终一致性

- 生产者、交换机、队列、消费者
- `OrderPaidEvent` 增加销量
- `OrderCancelledEvent` 恢复库存
- 消费者幂等
- 消息失败重试
- 死信队列

目标：把部分跨服务同步 RPC 改造成异步事件。

### 6. 本地消息表 / Outbox

- 本地事务成功但 MQ 发送失败怎么办
- 为什么直接 `tx.Commit()` 后发 MQ 不够可靠
- 本地消息表如何和业务表同事务写入
- 后台任务如何扫描待发送消息

目标：理解本地事务和消息投递之间的一致性缺口。

### 7. Binlog / CDC

- MySQL Binlog 是什么
- MQ 业务事件和 Binlog 数据变更事件有什么区别
- Binlog 适合同步 ES、报表、审计，不适合替代核心业务事件

目标：知道什么时候用 MQ，什么时候用 Binlog。

### 8. Redis 高并发缓存

- 缓存穿透、击穿、雪崩
- 热点 key
- 缓存一致性
- Redis 计数
- 分布式锁什么时候能用，什么时候不要乱用

目标：能给商品详情、库存、销量、点赞计数设计合理缓存方案。

### 9. 互动模块实战

互动模块不是边角功能，后面要单独实战一轮。

- 多级评论如何建模
- `parent_id` 和 `root_id` 的区别
- 评论列表如何避免 N+1 查询
- 点赞如何防重复
- 点赞计数如何维护
- 高并发点赞如何优化

目标：能独立讲清楚评论树、点赞幂等、计数一致性，这块面试很常见。

### 10. 前端端到端联调

- H5 / 小程序 / 管理后台如何适配后端接口
- token 如何贯穿前后端
- 响应结构如何统一
- API Fox、页面、数据库、日志如何一起验证

目标：把后端链路真正接到前端页面上跑通。

## 学习节奏

每轮保持这个顺序：

```text
先画流程
再定接口
再写 service 逻辑
再手动验证
最后记录文档
```

不要一次性做完整系统。每轮只抓一个核心知识点。
