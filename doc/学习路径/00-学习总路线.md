# 学习总路线

这份文档是后续学习的主线锚点。

每次不知道下一步做什么时，先看这里，再结合当前代码状态决定下一轮的小任务。

## 当前进度

已经完成或基本跑通：

- 商品列表：分页、搜索、分类、价格区间、排序、库存筛选
- 商品详情：Redis 缓存、空缓存、缓存失效
- 购物车：加购、重复商品合并、更新数量、列表汇总
- 订单预览：从购物车选择商品并计算总价
- 从购物车创建订单：创建订单、保存商品快照、扣库存、删除购物车
- 取消订单：状态流转、恢复库存、失败补偿
- 支付成功加销量：支付回调后通过 MQ 异步增加商品销量
- 销量补偿：`sales_status=2` 的订单可手动触发补偿
- 补偿任务健壮性：可恢复超时卡住的 `sales_status=3` 同步中订单
- 订单状态机整理：已梳理主状态流转，并把支付、取消接入状态合法性校验
- 订单超时未支付自动取消：扫描超时待支付订单，自动取消并恢复库存
- 补偿任务入口升级：已把手动补偿能力整理为 order-service 后台任务入口
- MQ 改造支付成功加销量：支付成功后发布 `OrderPaidEvent`，由消费者异步增加销量，并用 `sales_status` 做幂等和补偿衔接
- MQ 验证与故障演练：已能在 RabbitMQ 控制台手动发布 `OrderPaidEvent`，并通过日志、队列指标、`order_info`、`goods_info` 验证真实消费链路

## 当前下一步

```text

用已经产生的 sales_status=2 失败订单做补偿恢复演练，
```

```text

也真实遇到了 goods-service 服务发现失败导致 IncreaseSales 超时的故障。
下一轮继续把 sales_status=2 的失败订单补偿回来，并验证重复消息不会让 sale 再加一次。
```

目标：

```text
能解释 MQ 消息已经 Ack 但业务失败为什么还要靠 sales_status 补偿；
能从 order-service 日志、goods-service 日志、RabbitMQ 队列指标、MySQL 数据判断链路卡点；
能说清楚 sales_status 防重复的边界，以及为什么下一阶段需要 Outbox / 更强消费幂等。
```

## 后续学习阶段

### 1. 订单超时未支付自动取消

- 定时扫描待支付订单
- 超时后自动取消订单
- 自动取消后恢复库存
- 多实例部署下如何避免重复取消

目标：把状态流转、库存恢复、补偿思想串起来。

### 2. 补偿任务入口升级

当前补偿是手动 API 触发，后续要升级为后台任务。

- 后台任务放在哪里启动
- 日志怎么观察
- 多实例部署时如何避免重复执行
- 手动触发入口是否保留给管理端排障

目标：理解后台任务和普通 HTTP 接口的职责区别。

### 3. MQ / RabbitMQ 最终一致性

- 生产者、交换机、队列、消费者
- `OrderPaidEvent` 增加销量
- `OrderCancelledEvent` 恢复库存
- 消费者幂等
- 消息失败重试
- 死信队列

目标：把部分跨服务同步 RPC 改造成异步事件。

### 4. 本地消息表 / Outbox

- 本地事务成功但 MQ 发送失败怎么办
- 为什么直接 `tx.Commit()` 后发 MQ 不够可靠
- 本地消息表如何和业务表同事务写入
- 后台任务如何扫描待发送消息

目标：理解本地事务和消息投递之间的一致性缺口。

### 5. Binlog / CDC

- MySQL Binlog 是什么
- MQ 业务事件和 Binlog 数据变更事件有什么区别
- Binlog 适合同步 ES、报表、审计，不适合替代核心业务事件

目标：知道什么时候用 MQ，什么时候用 Binlog。

### 6. Redis 高并发缓存

- 缓存穿透、击穿、雪崩
- 热点 key
- 缓存一致性
- Redis 计数
- 分布式锁什么时候能用，什么时候不要乱用

目标：能给商品详情、库存、销量、点赞计数设计合理缓存方案。

### 7. 互动模块实战

互动模块不是边角功能，后面要单独实战一轮。

- 多级评论如何建模
- `parent_id` 和 `root_id` 的区别
- 评论列表如何避免 N+1 查询
- 点赞如何防重复
- 点赞计数如何维护
- 高并发点赞如何优化

目标：能独立讲清楚评论树、点赞幂等、计数一致性，这块面试很常见。

### 8. 前端端到端联调

- H5 / 小程序 / 管理后台如何适配后端接口
- token 如何贯穿前后端
- 响应结构如何统一
- API Fox、页面、数据库、日志如何一起验证

目标：把后端链路真正接到前端页面上跑通。

### 9. 秒杀高并发专题

这部分对应电商系统最极端的流量场景，不要和普通下单混在一起学。

- 秒杀服务为什么要独立部署
- 网关限流、令牌桶、服务熔断分别挡在哪一层
- Redis Lua 为什么能保证单 key 扣库存原子性
- Redis 扣库存和 MySQL 库存如何最终一致
- 常规下单用数据库条件更新，秒杀场景用 Redis Lua，两者边界是什么

目标：能讲清楚防超卖、削峰、限流、服务隔离和最终一致性。

### 10. ES 商品搜索专题

商品搜索后面要从数据库 `LIKE` 升级到搜索引擎方案。

- Elasticsearch 倒排索引是什么
- 商品名称、分类、描述如何做多字段加权
- 高亮、搜索建议、热词推荐怎么做
- MySQL 商品数据如何同步到 ES
- ES 和 MySQL 数据延迟时，业务怎么兜底

目标：能设计商品搜索链路，并知道 ES 不是主库，只是搜索读模型。

### 11. 网关治理与安全

现在项目里有 `gateway-h5` 和 `gateway-admin`，后面要补齐真实网关治理思想。

- H5 网关和 Admin 网关为什么要分开
- JWT 鉴权、Refresh Token、主动失效怎么设计
- Admin 的 RBAC、操作审计、IP 白名单怎么做
- Kong / Nginx / GoFrame 网关各自适合放什么能力
- 限流、灰度、黑白名单应该放网关还是业务服务

目标：能说清楚网关不是简单转发，而是流量入口治理层。

### 12. 链路追踪 / 监控 / 日志

微服务调用链变长后，不能只靠终端日志定位问题。

- TraceID 如何从 HTTP Header 传到 gRPC Metadata
- Jaeger 中 Trace 和 Span 的关系
- 日志里为什么要带 trace_id、user_id、order_number
- Prometheus 采集哪些业务指标：QPS、错误率、下单成功率、支付成功率
- Grafana 看板和告警怎么帮助排障
- Loki 日志聚合和本地日志文件有什么区别

目标：能从一次 API 请求定位到跨服务调用链、关键日志和指标异常。

### 13. 分库分表专题

这部分放到订单链路稳定之后再学，不提前复杂化当前代码。

- 为什么订单表最容易变成大表
- 按 user_id 分库、按时间分表分别解决什么问题
- Snowflake 全局 ID 为什么重要
- 分库分表后为什么跨库 JOIN 变困难
- 冗余字段和业务层组装如何替代 JOIN
- 深分页为什么会变慢，游标分页怎么解决

目标：能讲清楚分库分表的收益、代价，以及它对查询设计的影响。

### 14. K8s 云端部署专题

这部分放在本地 Docker Compose 跑通后再学。

- Docker Compose 和 Kubernetes 的职责区别
- Deployment、Service、ConfigMap、Secret 分别是什么
- HPA 自动扩缩容依据哪些指标
- 灰度发布和滚动发布怎么做
- 线上配置、日志、健康检查、重启策略怎么设计

目标：能把本地微服务迁移到云端部署，并知道线上运维最小闭环是什么。

### 15. 用户鉴权专题

这部分用于补齐用户服务和网关鉴权的面试表达。

- JWT 的签发、解析、过期和刷新
- `user_id` 为什么从 token 里取，而不是前端传
- 微信小程序登录 / OAuth2.0 登录流程
- Refresh Token、主动失效、黑名单怎么设计
- H5 网关如何把用户身份透传给后端服务

目标：能讲清楚用户身份从登录到跨服务传递的完整链路。

### 16. 支付与退款专题

这部分放在订单状态机和支付回调稳定之后再学。

- 微信支付发起支付、回调验签、交易号记录
- 支付回调为什么必须幂等
- 订单状态和支付状态如何避免冲突
- 退款接口、退款回调、退款状态流转
- 支付成功后加销量、取消订单恢复库存、退款后是否回滚销量

目标：能讲清楚支付不是简单改订单状态，而是一套外部回调驱动的状态一致性问题。

### 17. 管理后台与 RBAC 专题

这部分用于补齐 GFast 管理后台和权限模型。

- GFast 后台如何组织菜单、接口和角色权限
- RBAC：用户、角色、权限、资源之间的关系
- 商品管理员、订单管理员如何做权限隔离
- 后台操作审计应该记录哪些字段
- 管理后台接口和 H5 接口为什么要分网关

目标：能把管理后台从“能用”提升到“权限边界清楚、可审计”。

### 18. Go 并发与排障专题

这部分不单独追求炫技，重点是能解释业务里为什么这样并发。

- goroutine 和 channel 的适用场景
- 订单创建里哪些任务可以并发，哪些必须串行
- context 超时取消如何避免 goroutine 泄露
- WaitGroup、错误收集、panic recover 怎么用
- 如何排查 goroutine 泄露

目标：能在业务代码里谨慎使用并发，并能解释风险和边界。

### 19. MySQL 优化专题

这部分放在订单、商品查询链路稳定之后再补。

- 索引设计：联合索引、唯一索引、覆盖索引
- `EXPLAIN` 怎么看查询是否走索引
- 慢查询日志怎么定位问题 SQL
- 分页查询为什么会慢，游标分页怎么优化
- 事务隔离级别、行锁、死锁和库存扣减的关系

目标：能讲清楚除了 Redis 缓存之外，数据库层还能怎么优化。

### 20. 压测与效果验证专题

这部分用于把“优化了多少”变成有证据的表达。

- 压测前要明确哪些指标：QPS、TP95、错误率、数据库连接数
- 如何设计压测场景：商品列表、商品详情、下单、支付回调
- 优化前后如何对比响应时间和资源消耗
- Redis 命中率、RabbitMQ 堆积、MySQL 慢查询怎么一起看
- 面试里如何说明“200ms 优化到 80ms”“MySQL 压力降低 60%”

目标：能用数据证明优化效果，而不是只说“用了缓存所以变快”。

## 文档知识点映射

这篇文章《秒杀、分库分表、全链路追踪：一个电商微服务的架构全拆解》可以作为后续架构专题清单，但不是当前执行顺序。

飞书文档《GoFrame微服务电商项目-学习大纲》和《社招参考简历&面试题》可以作为知识点覆盖清单。它们提醒我们：除了主业务链路，还要补齐用户鉴权、支付退款、后台权限、Go 并发、MySQL 优化、压测验证这些面试高频点。

当前策略：

```text
先把订单主链路和一致性吃透
再学 MQ / Outbox / Binlog
再补 Redis 高并发、秒杀、ES、网关治理、链路追踪、分库分表、K8s
最后补面试专项，保证能把做过的功能讲清楚
```

不要被全景文章或面试题清单带跑。它们适合用来检查知识面，不适合作为当前每一轮的开发顺序。

## 学习节奏

每轮保持这个顺序：

```text
先画流程
再定接口
再写 service 逻辑
再手动验证
最后记录文档
```

不要一次性做完整系统。每轮只抓一个核心知识点。
