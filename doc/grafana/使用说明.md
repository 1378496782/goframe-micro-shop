# Grafana看板与动态阈值告警使用说明

## 1. 概述

本文档提供了Go微服务系统的Grafana监控看板和动态阈值告警配置的使用说明。通过导入本文档提供的配置文件，您可以快速建立服务监控体系，实现基于历史数据的智能告警。本文档适用于基于GoFrame框架开发的微服务项目，已根据项目中实际定义的Prometheus指标进行了优化配置。

### 1.1 功能特性

- **服务性能监控**：HTTP请求统计、响应时间、错误率、资源使用情况
- **动态阈值告警**：基于历史数据自动计算告警阈值，适应业务波动
- **业务指标监控**：订单量、库存等核心业务指标的异常检测
- **可定制性**：支持根据业务需求调整监控指标和告警规则

### 1.2 文件结构

```
doc/grafana/
├── dashboards/
│   └── go-service-monitoring.json  # 服务监控看板配置
├── alert-rules/

```

## 2. 前提条件

- Grafana 8.0+版本（动态阈值功能需要8.0或更高版本）
- Prometheus作为数据源
- 项目已通过`utility/metrics`包实现Prometheus指标导出

## 3. Grafana看板导入

### 3.1 登录Grafana

1. 访问Grafana Web界面（默认地址：http://localhost:3000）
2. 使用管理员账号登录

### 3.2 配置Prometheus数据源

1. 点击左侧菜单的「Configuration」→「Data sources」
2. 点击「Add data source」
3. 选择「Prometheus」
4. 在URL字段输入Prometheus服务器地址（如：http://prometheus:9090）
5. 点击「Save & Test」确认连接成功

### 3.3 导入监控看板

1. 点击左侧菜单的「+」→「Import」
2. 点击「Upload JSON file」
3. 选择 `doc/grafana/dashboards/go-service-monitoring.json` 文件
4. 在「Options」部分：
   - 选择已配置的Prometheus数据源
   - 可以修改看板名称和UID（建议保留默认值）
5. 点击「Import」完成导入

## 3.4 与项目集成说明

### 3.4.1 确认指标暴露
本项目已通过以下方式暴露Prometheus指标：

1. **指标注册**：`utility/metrics/metrics.go` 中的 `RegisterHTTPHandler` 函数已注册 `/metrics` 端点
2. **中间件集成**：`utility/metrics/middleware.go` 中提供了两个中间件：
   - `MetricsMiddleware`：记录HTTP请求指标
   - `ErrorMetricsMiddleware`：记录错误指标
3. **业务指标**：`utility/metrics/business.go` 中提供了业务指标记录函数

### 3.4.2 验证指标采集
1. 启动项目服务
2. 访问 `http://[服务地址]:[端口]/metrics` 确认指标正常暴露
3. 在Prometheus中执行查询验证数据采集

## 4. 动态阈值告警配置

### 4.1 配置告警通知渠道

1. 点击左侧菜单的「Alerting」→「Notification channels」
2. 点击「New channel」
3. 根据需要配置通知方式（如Email、Slack、Webhook等）
4. 点击「Save」保存通知渠道

### 4.2 手动配置动态阈值告警

> **注意**：由于Grafana 8.0+的动态阈值告警不支持直接从YAML导入，需要在UI中手动配置。以下是配置步骤：

1. 进入导入的「Go服务监控看板」
2. 点击需要配置告警的面板（如「HTTP请求速率」面板）
3. 点击面板标题 →「Edit」
4. 切换到「Alert」选项卡
5. 点击「Create Alert」
6. 在「Conditions」部分，点击「Select Query」
7. 选择「Dynamic Threshold」
8. 配置动态阈值参数：
   - **Learning period**：学习周期（建议7d）
   - **Model Sensitivity**：模型敏感度（建议Medium）
   - **Detect Mode**：检测模式（上升/下降/双向）
   - **Bound**：边界（上限/下限/双向）
9. 配置「Evaluate every」（评估频率）和「For」（持续时间）
10. 在「Notifications」部分，添加之前配置的通知渠道
11. 点击「Save」保存告警配置

### 4.3 根据YAML文件批量配置告警规则

对于`dynamic-alerts.yml`中的规则，建议按照以下方式批量配置：

1. 为每个规则创建对应的告警条件
2. 参考YAML中的注释设置学习周期、敏感度等参数
3. 根据业务重要性调整严重级别和通知方式

## 5. 动态阈值参数调优指南

### 5.1 学习周期设置

- **7-14天**：推荐值，能够捕获完整的工作日/周末模式
- **30天**：适用于有明显月度波动的业务
- **1天**：仅用于快速测试，不建议生产环境使用

### 5.2 敏感度调整

- **Low**：减少误报，但可能错过一些异常
- **Medium**：平衡误报和漏报，适用于大多数场景
- **High**：提高异常检测灵敏度，适用于关键业务

### 5.3 检测模式选择

- **Above**：仅检测值高于历史模式的情况（适用于错误率、响应时间等）
- **Below**：仅检测值低于历史模式的情况（适用于请求量、成功率等）
- **Above & Below**：双向检测（适用于需要严格监控的业务指标）

## 6. 常见问题与故障排除

### 6.1 动态阈值不生效

- 检查Grafana版本是否为8.0+（动态阈值功能的最低版本要求）
- 确认学习周期已完成（首次配置需要等待学习周期结束）
- 验证Prometheus数据源是否正确配置且有足够的历史数据
- 检查Prometheus是否已配置正确的抓取配置

### 6.2 告警频繁触发（误报）

- 增加动态阈值的敏感度设置（从High调整到Medium或Low）
- 延长「For」参数的值，要求异常状态持续更长时间才触发告警
- 检查是否存在数据采集异常或服务重启导致的指标波动

### 6.3 告警未触发（漏报）

- 降低动态阈值的敏感度设置（从Low调整到Medium或High）
- 缩短「For」参数的值，允许短暂的异常状态触发告警
- 检查指标查询表达式是否正确，确保能准确反映实际情况

## 7. 最佳实践

- **逐步部署**：先在非生产环境测试动态阈值告警，调整参数后再应用到生产环境
- **定期审查**：每周审查告警触发情况，根据实际业务模式调整参数
- **设置分组**：根据服务重要性和功能域对告警进行分组管理
- **渐进式配置**：先配置关键指标的告警，再逐步扩展到次要指标
- **告警分级**：根据业务影响设置不同级别的告警和通知渠道
- **与现有代码协同**：确保业务代码正确调用 `business.go` 中的指标记录函数，在新功能开发时，同步更新监控指标和看板配置

## 8. 现有指标说明

### 8.1 系统性能指标
项目中已实现的核心指标包括：

1. **HTTP请求计数** (`http_requests_total`)
   - 标签：`method`, `path`, `status`
   - 描述：记录所有HTTP请求的数量

2. **HTTP请求延迟** (`http_request_duration_seconds`)
   - 类型：直方图
   - 标签：`method`, `path`, `status`
   - 描述：记录请求处理时间分布

3. **服务错误计数** (`service_errors_total`)
   - 标签：`error_type`, `service`
   - 描述：记录服务产生的各类错误

### 8.2 业务指标
项目中已实现的业务指标包括：

1. **订单创建计数** (`business_order_create_total`)
   - 标签：`status`
   - 描述：记录订单创建情况

2. **订单成功率** (`business_order_success_ratio`)
   - 类型：Gauge
   - 描述：记录订单处理的成功比例

3. **产品库存** (`business_inventory_count`)
   - 标签：`product_id`
   - 描述：记录各产品的当前库存

## 9. 维护与更新

- 当服务架构变更时，及时更新监控指标和告警规则
- 随着业务发展，定期扩展监控维度和指标
- 关注Grafana版本更新，利用新功能优化监控体系
- 记录告警触发历史，分析系统稳定性趋势
- 本配置文件与项目中的`utility/metrics`包完全匹配，修改指标定义时请同步更新相关配置
